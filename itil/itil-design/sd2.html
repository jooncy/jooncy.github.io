# 第三章：服务级别管理

# 概述

# IT 与业务认知的差异

  • 价值展现
    • 业务觉得不存在的IT最好，因为找IT的时候都是发生问题的时候
    • 领导觉得IT就是修电脑的，人事觉得带电的都归IT管理
  • 故障处理
    • 运维花了一个小时修复了问题，还挺有成就感
    • 业务觉得好慢啊，影响了收益
  • 服务形式
    • 重启，重装... 业务觉得你技术不行

# 定义服务级别管理

  • 服务级别管理
    • 协商SLA
    • 确保满足SLA
    • 确保所有IT服务管理流程、OLA及基础合同对服务级别目标是否合适的
      • SLM剪口和报告服务级别，保存阶段性客户回顾
  • 目标
    • 通过持续的循环：统一、监控、报告，维护和改进服务质量
    • 消除差的服务（比如服务管理公司一个禅道、Jitamin）
    • 改进IT和客户之间的关系

# 定义目标

  • Objectives
    • 定义文档、监控、度量及回顾提供的IT服务级别
    • 提供改进业务和客户之间的关系及沟通
    • 确保定义了明确的和可衡量的IT服务目标
    • 监控和改进客户对提供服务的满意度
    • 保证IT和客户对服务级别有清楚的期望
    • 确保主动的度量，来改进服务交付的级别

# 术语

# 服务级别协议（SLA）

  • IT服务提供者和客户之间的协议
  • 面熟IT服务
  • 证明服务级别目标
  • 明确IT服务提供者和客户的职责
  • 一个单独的LSA可能覆盖多个IT服务或多个客户

# 服务级别需求（SLR）

  • 客户对IT服务的需求
  • 基于业务目标
  • 用于协商服务级别目标
  • 清除差的服务
  • 改进IT和客户之间的关系

# 支持合同（UC）

  • IT服务提供方与外部供应商就某一也定服务项目的提供与支持所签订的协议

# 操作级别协议（OLA）

  • IT服务提供者和统一组织中另一部分之间的协议
  • 支持IT服务提供者为客户提供服务
  • 定义提供的货物或者服务
  • 举例：服务台和一个解决事故的支持组

# 服务级别流程

# 服务级别管理流程（SLM）

  • 初始计划活动

  • 计划监控性能

  • 建立初步的服务

  • 支持合同UC和运营界别协议OLA

# 流程实施

总结一句话，先干活，然后签协议

  • 制作服务目录
  • 实施客户期望管理
  • 规划SLA结构
  • 切丁SLR及草拟SLA
  • 指定SLA
  • 与客户进行协商
  • 建立检测能力
  • 评审UC和OLA
  • 定义报告和评审程序
  • 发布LSA

# 流程运转监控

  • 检测和宝宝
  • 服务评审会
  • 服务改进计划
  • SLA/UC/OLA的维度

# 服务级别协议

# SLA结构

  • 基于服务
    • 涵盖一个服务
      • 举例：一个对所有用户的E-Mail服务
  • 基于客户
    • 和一个单独客户组的协议
    • 客户的所有需求
  • 对级别
    • 组合的层次
      • 举例：公司，客户，服务

# 关键性能指标（KPI）

  • SLA覆盖的服务条目
  • 所有SLA包括的支持合同和OLA
  • 客户感知统计
  • 服务级别完成情况
  • IT成本
  • 服务违背的严重程度
  • 监控所有KPI是服务级别经理的职责

# 服务级别经理角色

  • 负责保证满足服务级别管理的目标
  • 职责
    • 感知业务需求变化
    • 保证当前和未来的客户服务需求被确认、理解并记录在SLA和SLR中
    • 协商和统一提供客户的服务级别（内部及外部）
      • SLA的正式文档

# 挑战及建议

  • 风险
    • 缺少准企鹅的客户和业务输入、包含及委托
    • 需要工具及资源管理和监控SLM协议
    • SLM变得管理，而不是俄日客户提供利益
    • 低接受，绕过SLM流程
    • 高期望级别，低感知
  • 挑战
    • 谁负责服务？和谁协商？
    • 总公司是否知道分公司的需求？
  • 建议
    • 授信从简单的、没有争议的地方开始
    • 定义需求、显示、可行、满足开支
