# 第三章：事故管理

# 概述

  • 事故：IT服务的无计划中断或IT服务质量的降低
  • 事故管理：负责管理所有事故生命周期的流程

# 目的目标

  • 尽快恢复正常服务运营（ASAP）

# 范围

  • 来自服务台、或通过实践管理接口的中断或者可能中断服务的任何实践
  • 来自技术员工

# 价值

  • 检测和解决事故的能力，减少业务停机时间，支持服务的高可用性
  • 事故管理包括确定业务优先级和按需求分配资源，可以将IT活动与实时业务优先级对齐
  • 确定服务可能的改进
  • 服务台可以再处理事故的过程中，确定额外的服务或者培训需求

# 原则与基本概念

# 业务影响范围

  • 时间范围
  • 事故模型
  • 重大事故

# 临时措施

  • 用于临时消除事故或问题的方法，可能来自于事故管理或问题管理。不是最终的结构性解决，而是为了尽快恢复服务的运行而采取的临时的解决办法。

# 变更请求

  • 由事故管理或者问题管理等流程发起，提交给变更流程，来结构化的永久解决问题

# 流程

  • 事故鉴定
  • 记录日志
  • 分类
  • 确定 优先级
  • 诊断
  • 升级
  • 关闭

# 记录与分类

# 事故日志记录

  • 使用工具支持

# 分类

  • 分类编码规则
  • 按级别分类
  • 定期回顾并不断完善

# 升级

  • 如果某个事故在规定的时间内不能由一线支持小组解决，需要更多有经验的和更高权限的人员参与进啦
  • 可能发生在事故解决过程的任何时间和任何支持

# 升级的方式

  • 职能型升级
  • 结构型升级

# 事故关闭

  • 由服务台负责
  • 检测内容
    • 合适分类并修真
    • 用户满意度调查
    • 事故文档
    • 是否记录问题
    • 正式关闭

# 挑战

  • 尽早发现事故（监控）
  • 要求技术人员同用户一样记录所有事故并鼓励用户使用自助web平台来完成
  • 问题与已知错误信息的有效性
  • 与配置管理系统的集成
    • CI 之间的关系
    • 一线支持时，参考CI的历史
  • 与服务级别管理流程的集成
    • 基于业务影响分配合适的优先级
    • 定义和执行升级流程
    • 帮助SLM定义合理的目标

# 事故经历与考核

# 事故经理角色

  • 职责
    • 推动事故管理流程的有效性和高效性
    • 管理事故支持团队
    • 监控事故管理的效果，并提供改进建议
    • 开发和维护事故管理系统及流程
    • 管理重大事故
  • 很多组织中，事故经理角色富裕服务台主管
    • 在大型组织中，有必要分开
    • 重要的是，事故经理被授予权利来管理突发的一线、二线和三线支持

# 关键指标

  • 事故总数
  • 事故不同状态的明细
  • 当天未完成事故总数
  • 重大事故数目和比例
  • 平均解决时间，按优先级排序
  • 按SLA解决事故的百分比
  • 每个事故的成本
  • 重开的事故数目
  • 一线事故解决率
