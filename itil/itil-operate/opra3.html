# 第四章：问题管理

# 问题概述

  • 问题 - 一个或多个事故的原因

  • 问题管理：负责管理所有问题什么周期的流程

  • 问题管理的目标

    消除引起事件的深层根源防止事故再次发生

# 业务价值

  • 与事故管理、变更管理流程一起，保证IT服务可用性和质量提高
  • 额外价值

    • IT服务的高可用性

    • 业务与团队的高生产力

    • 减少因为临时解决方案的开支

    • 减少旧货和重复解决事故的成本

# 问题管理

  • 被动问题管理

    • 找出导致以前的事故发生的根本原因
    • 找出解决措施或者纠正建议

  • 主动问题管理

    • 通过找到基础设施中薄弱环节来阻止事故的再次发生
    • 提出消除这些薄弱环节的建议

# 事故管理和问题管理

  • 突发事件管理
    • 恢复服务到协商之上的级别
    • 使用规避措施
  • 问题管理
    • 诊断如法事件的根本原因
    • 识别最终解决方案
    • 可能花费更长的时间

# 事故和问题之间深层次的联系

核心要素

  • 事件
  • 事故
  • 问题管理
  • 已知错误
  • 变更

# 问题管理流程

  • 问题检测、记录日志
  • 问题分类和优先级
  • 问题调查诊断
  • 临时措施
  • 已知错误
  • 重大问题回顾
  • 问题关闭

# 临时措施与已知错误

  • 临时措施

    在事故或问题还没有完全的解决办法可用时，降低或消除其影响的方法

  • 已知错误

    已经有文档记录的根本原因和临时的问题

  • 已知错误数据库

    存放已知错误记录的数据库，有问题管理创建，供事故和问题管理使用

# 问题调查诊断工具

# 可用技术和工具

  • 按实际顺序分析

    例：版本迭代后出现问题，找最近的时间节点作为排查对象，做过哪些修改

  • 头脑风暴/鱼骨图

    个人比较推荐这种方法，集思广益 & 批判性思维，颠覆性思考，完全使用思维导图的方式分析问题的可能性

  • 帕累托分析

    适合月问题总结

  • 5-Ways

    美国军方提供的解决问题方式，深入思考. 多问为什么

问题管理经理与度量指标

问题回顾一般一个月以内

# 协调所有问题管理活动

  • 职责
    • 联络所有问题解决小左，保证SLA模板前提下迅速解决问题
    • 已知错误数据库的属主和保护者，负责已知错误的挂历和搜索算法
    • 正式关闭所有问题记录
    • 联络厂商，承包商等确保第三方文爱吃解决问题相关合同
    • 安排、运行、记录文档、跟踪重大问题回顾相关活动

# 度量指标

  • 某个阶段内记录问题的总数
  • 在SLA范围内，解决问题的百分比
  • 长处约定解决的问题数目和百分比
  • 尚未解决的问题和发展趋势
  • 解决问题的平均成本（量化，金额）
  • 重大问题的数目及重大问题回顾执行的百分比
  • 增加到KEDB的已知错误数目
  • KEDB的准确度

# 问题管理的挑战

  • 问题管理流程中重大的挑战是确保问题管理好事件管理有着真实的接口和协同工作的

    • 频繁发生（TOP-N中前三，看精力和情况而定）
    • 影响重大

  • 问题解决人员的技术和能力来识别故事的真是根本原因，有时候也是 一种挑战

  • 确保问题管理能够使用所有的只是和服务资产，而且配置管理资源对调查和解决问题也是可用的

  • 确保不断对技术人员进行培训，内容涵盖对其工作技术内容，以及他们支持的服务和使用的流程的业务培训

  • 确保业务影响所有解决问题的人员立即

  • 如果用于记录事故和问题的工具不同，将事故与问题关联的能力成为一项挑战

  • 二线与三线能够在问题支持活动中建立良好的工作关系的能力
