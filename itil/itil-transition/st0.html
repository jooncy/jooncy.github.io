# 第一章：变更管理

# 理解变更和风险

# 变更的两个原因

  • 被动变更
    • 修正基础架构中的错误
    • 适应改变了的环境
  • 主动变更
    • 寻找业务授予，如节约成本、提升服务、增加支持的有效性，降低难度
  • 为什么变更需要被管理？
    • 风险最小化
    • 影响和终端严重程度最小化

关键提示

所有变更都有风险，80%的问题都是变更导致的

# 定义变更管理

  • 变更管理
    • 控制所有变更的生命周期
    • 在最少的终端下实现现有收益的变更
  • 目的
    • 相应客户业务变更需求，实现价值最大化，减少事故，业务中断和返工
    • 相应业务和IT需求变更，是的服务器和业务需求一致

# 变更的价值

  • 提高可靠性和业务连续性，对任何组织的成功和生存而言都是最基本的
  • 变更管理推动业务
    • 确定优先级并听取相应业务和客户变更意见
    • 实现变更满足客户认可的服务需求，同时优化成本
    • 减少失败的变更，以及服务中断，缺点和返工
    • 按照时间表及时提供变更
    • 在服务生命周期中跟踪变更
    • 提供更好的质量，事件和成本的估算

# 定义变更的范围

# 变更触发

  • 变更可以被组织中的任何人触发
  • 服务的变更只能被如下流程/只能提出
    • 服务战略和业务关系管理流程 - 战略变更
    • 一个建议的开发项目的服务设计
    • 持续服务改进流程，用于改进效力和效率
    • 服务级别管理，用于改进服务级别
    • 服务操作，用于解决服务中心检测的已知错误
    • 外部厂商提交的变更请求
    • 遵照法律和政府规定

# 变更管理术语

# 变更流程

# 变更请求（RFC）

  • 执行变更的一个正式提议，包含了建议变更的细节
  • 不是变更记录，也不是变更本身

# 变更顾问委员会（CAB）

  • 对变更进行评估、有限级确定、日程安排负责的小组

# 紧急变更顾问委员会（ECAB）

  • CAB的子集，决策高影响度的紧急变更

# 变更类型

# 正常变更（Normal）

  • 组织特定的类型
  • 具体类型决定不同的评估，不紧急，但是影响大

# 标准变更（Standard Change）

  • 预授权的变更并带有具体步骤，不紧急，不重要的变更（bug修复）

# 紧急变更（Emergency Change）

  • 事件不足够来作为正常变更处理（故障）
  • 如果可用尽量以正常变更处理

# 变更管理流程

变更流程

如下图ITIL提供的变更管理流程，思维转换一下其实就是个上线流程，其中包含上线申请、上线审批、业务协调、上线计划，再看这图就会清晰的多，完全套用概念会模糊

案例

目前公司使用WIKI作为上线流程的确认过程，在某个工作空间中有一个上线确认单用于确认各个环节的交互，以及变更信息用于确认和回顾。 当产品，测试，技术，运营者几个节点确认后，即可确认上线，运维操作自动化部署

# 变更评估的7R

  • 谁发起（Raised）变更？

    • 运营，产品，技术总监

  • 变更的原因是什么（Reason）？

    • 版本优化，bug修复，重大事故回滚

  • 变更的回报（Return）是什么？

    • 新业务抢占市场份额，加强用户粘性

  • 变更包含的风险（Risk）是什么？

    • 数据库，缓存，本次新增了xxx，将给服务器带来压力

  • 交付变更，需要哪些资源（Resource）？

    • 服务器申请，mysql申请，MQ等

  • 谁负责（Respinsible）构造、测试、实施变更？

  • 这个变更和其他变更直接的关系（Relationship）是什么？

    • 业务拓展、业务加强

# 批准变更（Authorize）

# 变更批准可以是

  • 角色
  • 一个人
  • 一组人员

# 大型企业

  • 尤其是地狱粉丝的企业中，需要一个高级别变更批准机构，如：全局变更委员会，或者管理委员会

# 批准的级别通常依赖于变更的属性

  • 组织可以根据预先设置的参数来确定批准级别
    • 预期的商业风险
    • 财务
    • 变更范围（嫩不、财务服务、外包服务...）

# 变更批准模型

# 变更经理和KPI

变更经理

负责接收记录，分配优先级协助发起者处理RFC，这句话有点拗口. 转换一下这个人做的工作是项目经理 主要负责项目进度的跟进，这个职位的沟通能力，细心程度需要高一些

# 职责

  • 主持CAB和ECAB会议

    • 为CAB会议组织所有RFC

      • 会期前将RFC地 议程提前交给CAB成员，以提前准备

    • 决定哪些人参加会与

      • 根据RFC，确定分配给谁
      • 考虑变更什么，需要哪些专家意见

    • 对所有紧急RFC着急ECAB会议

  • 考虑CAB及ECAB的建议后，批准满足要求的变更

  • 监控KPI

# 度量指标

  • 变更管理的关键性能指标（KPI）
    • 满足实现商定的质量、成本及交付时限需求的变更数目
    • 变更的收益（ROI中约定）
    • 减少中断的数目
    • 减少未批准变更的数目
    • 减少RFC积压

# 变更管理的挑战

# 不佳的变更管理

  • 未授权的变更
  • 计划外的中断
  • 低的变更成功率
  • 大量的紧急变更
  • 延迟的项目实施

# 变更管理-挑战

  • 人员忽略这个流程
  • 不充分的配置管理
  • 业务方的压力“去做”（说这话的产品很糟糕）
  • 缺乏“标准变更”
  • 大型组织的规模影响
