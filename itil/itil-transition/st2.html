# 第三章：部署管理

# 发布和部署管理概述

# 发布管理和部署管理的职责

  • 发布管理
    • 负责将版本转移到测试和现场环境，包括计划和控制
    • 确保现场环境的版本以及发布的组件版本
  • 版本（release）
    • 硬件、软件、文档流程及其他组件的集合，用于实现一个或多个批注的IT服务变更，该变更作为一个实体管理、测试和部署
  • 部署
    • 负责将新的或者变更的硬件、软件、文档、流程等移动到生产环境的活动

# 价值

  • 发布和部署管理的价值
    • 快速交付变更，降低成本，较少风险
    • 保证客户和用户可以按照业务目标来使用心得或者变更的服务
    • 提升业务变更，服务组，厂商和客户的服务一致性
    • 有助于实现服务转换过程中的审核需求

# 目标

  • 创建清晰地，全面的版本和部署计划
  • 按照日程，成功构建、安装、测试和部署版本包到部署组成或者目标环境
  • 确保新服务或者变更服务及其支撑系统可以交付认可的服务需求
  • 保证对产品服务、操作和支持组织的魏雨琦影响最小
  • 确保知识转移
    • 使操作和支持团队可以依照需要的保障和服务级别有效和搞笑提供支持及维护
    • 使客户和用户可以正确使用服务
    • 验证客户、用户和服务管理人员是否对服务转换事件及输出满意（如：用户文档、培训等）

# 发布和部署管理流程

# 发布

  • 计划和准备发布
  • 运维发布到TEST
  • 业务测试（是否满足预期）
  • 运维部署生产/预发
  • 回顾/关闭

# 基本概念

  • 发布规模
    • 重大发布
    • 小的发布
    • 紧急发布
  • 发布和部署模型
  • 进入标准和退出标准

    • 设计规范

# 发布单元

  • 通常一起发布的IT服务组件，发布单元通常包括执行有用功能的充足组件。例如，一个发布单元可以是台式机，包括软硬件，许可，文档等不同的发布单元
  • 可以是完整的发布应用，包括IT运营程序和用户培训。

# 发布包

  • 一个或多个相关的发布
  • 可用包括硬件，软件，功能，功效，文档，培训等..

测试实践

测试环境部署， 其实是整套的环境分层。分别在开发环境，测试环境，预发环境分别进行测试验证，以保障产品上线的安全性与完整性。

# 环境分层

  • 开发环境
  • 测试环境
  • 预发环境
  • 线上环境

# 价值

  • 保障多版本一致性

  • 降低产品上线风险

  • 手动备份数据（Jenkins 自动备份）
  • 对变更的配置保留证据（SQL语句，申请单签字）

# 应急方案

  • 备件
  • 应急响应预案

# 部署新版本

  • Big Bang：新的或者变更的服务在一个操作中部署到所有用户范围。这种选项经常用于重点烤炉应用程序变更和整个组织服务一致性的情况
  • Phased：服务最初部署在用户应用的一部分，随后部署操作重复执行，按照发布计划时间表应用于用户的后续部分。

# 推 VS 拉

  • 推 使用情景是从中央发布到目标站点（Jenkins 一组一组发布）
  • 拉 让软件包在中央位置可用，单用户可自由把软件拉到自己的位置暗转

# 知识管理

# 目的目标

  • 确保正确的信息，帮助决策侧和运营支持
  • 确保组织在真个服务生命周期中可以通过可靠地、安全的信息、数据来改进决策管理
  • 知识管理贯穿整个服务生命周期，服务转换中对知识管理仅作基本的概念讲解

# 业务价值

  • 知识管理在服务转换中尤为重要，原因是转换时需要相应的知识来保障。
  • 距离说明成功的转换以来知识管理
    • 用户，帮助台等理解新的换变更的服务包括用于错误的知识有助于他们各自的岗位工作。
    • 了解服务的使用包括之前版本
    • 在转换的同事建立可接受级别的风险和信任级别，例如度量，正确了解和相应结果，并确保它们。
    • 知识管理是一种资产，个人或团队可以共享数据，信息和知识。
